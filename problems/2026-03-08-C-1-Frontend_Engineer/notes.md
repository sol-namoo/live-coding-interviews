1. 리스트 불러와서 렌더링
2. 상세보기 버튼 클릭 시 상세 정보 fetch 후 하단 렌더
3. 불필요한 리렌더링 방지 (콘솔에 `This error should not be printed`가 찍히지 않게)

## 내가 했던 접근 (복기)

### Task 1

- 비교적 빠르게 목록을 화면에 띄움.
- 당시에는 “일단 빨리 보이게” 만드는 데 집중해서, 구현 위치(Effect/State 흐름)보다 결과 중심으로 접근.

### Task 2 (가장 지체된 구간)

- Card에서 `onClick={() => {}}` 형태를 보고, 비어 있는 콜백 내부에 상세 fetch를 넣으려다 구조가 꼬임.
- 나중에 Card 내부를 확인하고, 실제로는 **Card가 콜백을 props로 받아  `onClick(item.id)` 형태로 상위 핸들러를 호출하는 구조**임을 파악.
- 이후 id 파라미터를 어디서 받아서 API를 호출할지를 다시 정리해야 했고, 그 과정에서 시간을 많이 사용.

### Task 3

- 처음에는 의도 자체를 이해 못 함.
- `This error should not be printed` 문구를 검색해서 로깅 로직을 찾았지만, 클래스 컴포넌트 형태인 것을 보고 당황하여 제대로 못 읽음.
- 막판에 힌트를 얻고 “리렌더 최적화(React.memo 등)로 해결하라는 뜻”임을 파악.

### Task 4

- 로딩 state를 두고 true/false 토글로 구현 가능했음.

## 해설(지금 생각하는 정상 흐름)

### 전체 흐름(지도)

- **목록**: 최초 1회 `getChallengeMiniInfoList()` → `list`에 저장 → 카드 렌더
- **상세**: 카드 버튼 클릭 → `id` 전달 → `getChallengeDetailInfo(id)` → `detail`에 저장 → 하단 렌더
- **로딩**: 상세 fetch 시작 시 `loadingDetail=true`, 종료 시 `false`

### Task 2에서 핵심이었던 것

- Card는 “fetch를 하는 곳”이 아니라 **id를 상위로 올려주는 프리젠테이션 컴포넌트**로 보는 게 자연스럽다.
- 즉 “상세 fetch + detail state”는 App 쪽(상위)에서 관리하는 게 깔끔하다.

## Task 3(리렌더 최적화) 관점에서 점검할 체크리스트
“어디서 불필요한 업데이트가 발생하는지”를 찾고 방지하는 문제였다.

1. **불필요한 state 변경이 있는가?**
   - 클릭 한 번에 여러 state가 연쇄적으로 바뀌면서 예상보다 많은 업데이트가 발생하지 않는지

2. **props가 매 렌더마다 새로 만들어지지는 않는가?**
   - `onClick={() => ...}` 같은 inline 함수가 하위로 내려가면서 매번 다른 레퍼런스로 전달되는지
   - 필요하면 `useCallback`으로 안정화

3. **컴포넌트 메모이제이션이 필요한가?**
   - `Card`가 list 아이템 단위로 렌더 비용이 크다면 `React.memo(Card)` 적용 고려

4. **key가 안정적인가?**
   - (이 문제에서는 의도되지 않았던 걸로 기억하지만) list 렌더에서 key가 index로 되어 있으면 컴포넌트 재사용이 꼬여 예상치 못한 update가 생길 수 있음

5. **Effect / Fetch가 불필요하게 재실행되는가?**
   - 목록 fetch가 클릭 때마다 재실행되거나, 상세 fetch가 같은 id로 중복 호출되는지

## 내가 아쉬웠던 점 / 다음에 개선할 점

1. 시간 안에 동작을 전부 구현해야 한다는 생각에 조급해서, **가정(이 코드/컴포넌트가 무엇을 책임지는지)을 확인**하는 걸 덜 했다.
2. Card 컴포넌트의 클릭 로직을 구현하는 데 급해서, onClick을 버튼의 onClick 속성으로 착각했고, 사실은 props라는 걸 인지하지 못했다. Task 2에서 막혔을 때, UI를 기준으로(의심하고 실제 Card.tsx 내부를 들여다보면서) **이벤트 → state → 렌더 흐름**을 차분히 다시 그리고 실제로 콘솔 로깅을 하면서 짚어갔어야 했다.
3. Task 3은 문구를 찾자마자 “최적화 문제”일 수 있다고 가설을 세우고, **어디가 재렌더를 유발하는지부터** 좁혀갔으면 더 빨랐을 것 같다. 최적화 문제라는 점을 확인하고 나서도, 일단 떠오르는 대로 'React.Memo 같은 걸 써야겠네요'라고 대답하지 않고 가능한 렌더링 최적화의 방식을 최대한 많이 떠올렸으면 좋았겠다.

## 다음에 같은 유형을 받으면 이렇게 한다

- 먼저 JSX로 화면을 그려보고, “클릭하면 무엇이 바뀌어야 하지?”를 기준으로 상태/이벤트 흐름을 파악한다.
- “이 로직이 어디에 있어야 자연스럽지?”(App vs Card)를 생각해 본다.
- 최적화(Task 3)는 코드 전체를 고치려 하기보다:
  - (1) state 변경 최소화
  - (2) stable props (useCallback)
  - (3) memoization (React.memo)
  가 가능한지 점검한다.
